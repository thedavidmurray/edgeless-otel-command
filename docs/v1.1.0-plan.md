# v1.1.0 Plan — Functional Foundations for Sharing

**Goal**: ship a version of the dashboard that a stranger can install on their machine, point at their Jaeger, and trust the data they see. Set up the foundation that future Winamp-style skin work plugs into.

## Scope

### 1. State + routing foundation
- Global `state` object: `{ jaegerUrl, services, serviceFilter, timeWindow, view, selectedTraceId }`
- Hash-based router: `#/overview`, `#/trace/<id>`, `#/service/<name>`, `#/anomalies`, `#/settings`
- Header: breadcrumb + back button + service/time selectors
- All panels subscribe to state; one render entry point dispatches by view

### 2. Drop hardcoded `SERVICE = 'edgeless-swarm'`
- Fetch traces from ALL services by default
- Service selector in header: `All / edgeless-swarm / claude-code / jaeger-all-in-one`
- Aggregation across services in overview metrics
- Per-service rollup card in service grid (traces/min, err rate, p95)

### 3. Real anomaly detection (replaces hardcoded HTML)
- **Phantom stall**: a non-leaf span whose children have all finished but the span itself stays open > `phantomStallMin` minutes
- **Ghost agent**: a known service has emitted zero spans in `ghostAgentMin` minutes
- **Loop detect**: `loopThreshold` spans of the same operation in `loopWindowSec` from one service
- Persistent anomaly log (last 100 events, stored to userData)
- Threshold configurable per anomaly type in settings

### 4. Configurable Jaeger endpoint + settings panel
- `~/Library/Application Support/edgeless-otel-command/settings.json`
- Settings UI accessible via header gear icon, also via `#/settings`
- Fields: Jaeger URL, refresh interval, anomaly thresholds, default theme, default service filter
- IPC: `settings:get`, `settings:set`, `jaeger:test` (test connection to a URL)
- `main.js` reads jaeger URL from settings at startup

### 5. First-run setup wizard
- On launch with no settings file: show full-screen modal wizard
- Steps: detect Jaeger (try localhost:16687 + 16686), prompt URL if none, test connection, pick default theme
- Persist to settings file
- Closeable; user can reopen via `#/settings`

### 6. Drill-down: trace detail view (`#/trace/<id>`)
- Fetch full trace via `/api/traces/<id>`
- Render waterfall: spans laid out by start time, indented by parent-child depth
- Click span → expand to show tags, logs, duration breakdown
- Header: trace ID, total duration, span count, root service, "Open in Jaeger UI" button
- Back to overview button

### 7. Drill-down: service detail view (`#/service/<name>`)
- All operations for that service with p50/p95/p99
- Recent traces filtered to this service
- Anomalies that have fired against this service

### 8. Click-everything (interactivity pass)
- Recent traces row → `#/trace/<id>`
- Op breakdown bar → filter overview to that op
- Service in service grid → `#/service/<name>`
- Anomaly card → `#/trace/<id>` of the offending span (when applicable)
- Latency heatmap row → filter to that op

### 9. Release engineering
- Bump version to 1.1.0 in package.json
- Build: `npm run build:mac` (arm64 + x64)
- `gh release create v1.1.0 --notes "..."` with .dmg + .zip artifacts
- Update README with screenshot of new dashboard

## Out of scope for v1.1.0 (future)

- Winamp-style full skin system (themes.css already has 6 skins; this is a placeholder for richer reskinning)
- Plug-in / panel module system (extract panels into separate files)
- Multiple Jaeger backends (tabs)
- Alert webhooks (Slack/Discord/Telegram)
- electron-updater auto-update
- TraceQL/PromQL raw query bar
- Trace JSON export
- Service map visualization

## Architecture notes for v2 skin work

To prep for Winamp-style skins, panels should eventually be modules with this interface:

```js
// panels/throughput.js
export const panel = {
  id: 'throughput',
  label: 'Throughput & Metrics',
  render(state, container) { ... },
  // future skin support
  layout: { defaultPosition: 'top-left', minSize: { w: 320, h: 240 } },
}
```

A "skin" then becomes: a CSS file + a layout JSON (which panels, where, sized how) + optional custom panels.

For v1.1.0 we keep panels inline but use named render functions so the eventual extraction is mechanical.
